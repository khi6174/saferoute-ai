# AGENTS.md — SafeRoute AI

## 0. Project Identity

SafeRoute AI is an AI-powered delivery safety decision-support MVP.

The system predicts parcel courier accident risk, explains why the risk is high, and recommends safer operational actions such as safer routes, delivery schedule adjustment, workload redistribution, delay recommendation, and rest recommendation.

This is not a simple monitoring app.  
This is not a simple delivery analytics dashboard.  
This is a safety-centered AI decision support system for parcel delivery operations.

Core message:

> We do not make delivery faster. We make delivery safer.

---

## 1. Agent Role

You are the coding agent for SafeRoute AI.

Your job is to build a realistic, demo-ready MVP suitable for an AI Rookie competition.

Prioritize:

1. Clear demo flow
2. Strong AI decision-making logic
3. Explainable risk output
4. Realistic mock data
5. Clean UI/UX for judges
6. Safe and privacy-conscious implementation

Do not overbuild.  
Do not create unnecessary enterprise complexity.  
Do not hide important logic inside vague helper functions.

The judging value comes from showing a clear chain:

```text
delivery situation → risk prediction → reason explanation → action recommendation
```

---

## 2. Development Mode

Work in small, reviewable steps.

Before editing:

- Inspect the existing repository structure.
- Identify the stack already being used.
- Reuse existing components, naming style, and folder patterns.
- Do not replace the whole project unless the repository is empty or clearly broken.

When uncertain:

- Prefer simple, deterministic MVP logic over complex unfinished architecture.
- Prefer mockable modules over real external API dependency.
- Prefer readable business logic over clever abstractions.

---

## 3. Expected MVP Flow

The MVP must support this 3-minute demo scenario:

1. Select or generate a demo delivery day.
2. Show courier list and current delivery status.
3. Select a high-risk courier.
4. Predict accident risk score.
5. Explain top risk factors.
6. Compare fastest route vs safer route.
7. Recommend schedule adjustment.
8. Show manager dashboard impact:
   - before risk score
   - after recommendation risk score
   - redistributed packages
   - delayed low-priority deliveries
   - recommended rest action

The MVP should make judges understand the product within 5 seconds of seeing the main screen.

If the repository already has a stack, follow it.

If starting from scratch, prefer one of the following:

### Option A — Fast MVP Web App

- Frontend: React + TypeScript + Vite
- Styling: Tailwind CSS
- Data: local JSON or generated mock data

- Streamlit

## 6. Product Rules

### 6.1 Risk Prediction

The system must calculate a risk score from 0 to 100.

Risk level mapping:

```text
0–39   Low
40–59  Medium
60–79  High
80–100 Critical
```

The score should consider at least:

- delivery count
- remaining delivery count
- total distance
- working hours
- delay minutes
- weather severity
- hill ratio
- villa density
- narrow road ratio
- night delivery
- parking difficulty
- fatigue score
- time pressure score

The risk score must be explainable.

Do not output only a number.

---

### 6.2 Explanation

Every risk prediction must include top risk reasons.

Each reason should include:

- factor name
- current value
- impact level
- human-readable explanation

Example:

```json
{
  "factor": "working_hours",
  "value": 9.2,
  "impact": "high",
  "message": "누적 근무 시간이 9시간을 넘어 피로 누적 위험이 큽니다."
}
```

Use Korean for end-user messages.

---

### 6.3 Safe Route Recommendation

Always compare at least two route options:

1. Fastest route
2. Safer route

Do not present the fastest route as automatically best.

Route scoring should consider:

- estimated time
- accident zone count
- hill sections
- narrow road sections
- school zone count
- parking difficulty
- weather impact
- night visibility risk

The safer route may take longer, but should reduce risk.

Example decision:

```text
최단 경로는 31분이지만 위험도 78점입니다.
안전 경로는 38분이지만 위험도 43점으로 낮습니다.
AI는 안전 경로를 추천합니다.
```

---

### 6.4 Schedule Recommendation

The system must recommend at least one action when risk is high or critical.

Possible actions:

- Reorder delivery sequence
- Redistribute packages to a lower-risk courier
- Delay low-priority deliveries
- Recommend rest
- Avoid high-risk region after rain/night
- Process hill/villa region earlier
- Send delay notice to customer

Every recommendation must include expected risk reduction.

Example:

```json
{
  "action": "redistribute",
  "message": "기사 A의 일반 배송 8건을 기사 B에게 분산합니다.",
  "expectedRiskBefore": 86,
  "expectedRiskAfter": 62
}
```

---

### 6.5 Real-time Alert

Real-time alerts must follow this format:

```text
위험 감지 → 원인 설명 → 권장 행동
```

Bad:

```text
위험합니다.
```

Good:

```text
위험도 82점: 강수 시작, 누적 근무 9시간, 남은 배송 38건으로 위험이 상승했습니다.
권장 행동: 다음 3건 배송 후 10분 휴식하고 급경사 구간을 우회하세요.
```

## 8. AI / Engine Implementation Rules

### 8.1 Initial MVP Engine

For MVP, deterministic scoring is acceptable and recommended.

Use weighted scoring first.

Do not pretend to have trained a real accident model if no real data exists.

Recommended modules:

- `fatigueModel`
- `riskScoring`
- `routeScoring`
- `scheduleRecommender`
- `explanationGenerator`

The code should be written so that a real ML model can replace the scoring function later.

---

### 8.2 Risk Score Formula

Initial formula example:

```text
risk_score =
  0.20 × workload_score
+ 0.20 × fatigue_score
+ 0.15 × time_pressure_score
+ 0.15 × weather_score
+ 0.15 × region_risk_score
+ 0.10 × route_risk_score
+ 0.05 × delay_score
```

Normalize final score to 0–100.

---

### 8.3 Fatigue Score

Fatigue score should consider:

- working hours
- total distance
- completed deliveries
- time since last break
- delay pressure

Example:

```text
fatigue_score =
  working_hours_component
+ distance_component
+ completed_delivery_component
+ break_component
+ delay_component
```

---

### 8.4 Explainability

Explanation generation must be linked to the score.

Do not hardcode generic explanations unrelated to actual values.

For every high contributing factor, generate a Korean explanation.

Example factors:

- `workingHours > 8`
- `remainingDeliveries > 40`
- `rainLevel > 0.5`
- `hillRatio > 0.6`
- `villaDensity > 0.6`
- `delayMinutes > 20`
- `lastBreakMinutesAgo > 180`

---

### 8.5 Future ML Extension

If implementing Python ML:

- Use synthetic data generator first.
- Train a baseline classifier or regressor only if time allows.
- Prefer LightGBM/XGBoost-style model design conceptually.
- If packages are unavailable, use scikit-learn RandomForest or GradientBoosting.
- Always expose feature importance if model is trained.

Do not make unsupported claims about real-world accident reduction.

Use “simulation result” wording.

## 10. Safety, Privacy, and Ethics

### 10.1 Privacy

Do not store real personal information.

Do not create fields for:

- resident registration number
- exact private home address
- phone number
- sensitive personal data

Use address labels like:

```text
관악구 빌라 밀집 구역 A
분당구 아파트 단지 B
강남구 상가 구역 C
```

---

### 10.2 Avoid Worker Surveillance Framing

The product must not look like a worker punishment system.

Frame all risky courier data as:

- safety support
- workload balancing
- rest recommendation
- accident prevention

Do not add ranking language such as:

```text
worst courier
bad performance
low productivity
```

Use:

```text
고위험 기사
지원 필요 기사
물량 조정 필요
휴식 권장
```

---

### 10.3 AI Responsibility

The AI must not make forced operational commands.

Use wording:

- 추천
- 권장
- 판단 보조
- 관리자 확인 필요

Avoid:

- 강제
- 명령
- 자동 배차 확정
- 책임 전가

Show disclaimer in the UI or README:

```text
SafeRoute AI는 사고 위험과 배송 조정안을 제안하는 의사결정 보조 시스템입니다.
최종 운영 판단은 현장 관리자와 기사에게 있습니다.
```


## 12. Testing Rules

Add tests for engine logic when possible.

Minimum test cases:

1. High workload increases risk.
2. Rain increases route risk.
3. Long working hours increases fatigue score.
4. High-risk courier receives at least one recommendation.
5. Safer route can be recommended even when slower.
6. Explanation includes actual high-impact factors.
7. Schedule recommendation reduces expected risk.
8. Low-risk courier should not receive aggressive delay recommendation.

Do not skip tests for risk scoring if the project has a test framework.

## 15. Definition of Done

A task is done only when:

- The app runs without crash.
- Demo data is visible.
- Risk score is calculated.
- Risk reasons are shown.
- At least two route options are compared.
- Schedule recommendation is generated.
- Manager summary shows before/after impact.
- UI text is understandable in Korean.
- README run instructions are updated.
- No fake claim is made about real accident reduction.
- No sensitive personal data is introduced.

---

## 16. Priority Order

Build in this order:

1. Data types
2. Mock data
3. Risk scoring engine
4. Explanation generator
5. Route scoring
6. Schedule recommender
7. Basic UI
8. Manager dashboard
9. README
10. Tests and polish

Do not start with authentication, database, deployment, or real map API.
Those are not necessary for the first MVP.

---

## 17. Forbidden Work for MVP

Do not spend time on:

- Real user login
- Real courier GPS tracking
- Real customer addresses
- Real payment or delivery company integration
- Full production database
- Complex microservices
- Overly advanced ML pipeline before demo logic works
- Automatic final dispatch command
- Worker performance ranking

---

## 18. Final Product Philosophy

Every feature must support this idea:

> 빠른 배송을 최적화하는 시스템은 많다.  
> SafeRoute AI는 안전한 배송 의사결정을 최적화한다.

If a feature does not help predict risk, explain risk, or recommend safer action, cut it from the MVP.
